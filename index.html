谈谈社交这回事

谈谈社交这回事

我是一个不太有朋友而且也没有什么交朋友的技巧的人。
作为技术宅，更是坚信只有技术才能创造价值。
写代码是硬功夫，代码漂亮完善性能好占用资源少，都是可以被衡量的，但是”社交“不能被衡量，不能像”花了2h学会了python web基础“一样有明确的时间消耗和产出结果。
大部分的时间都是在钻研技术，写代码看书，打打游戏，跟人打交道的时间很少，沟通都是追求简单直接。

最近一段时间发现，社交已经成为了目前的一个瓶颈。
首先工作内容更多的转变到沟通方面：跟团队成员、其他职能部门、淘宝等合作公司的沟通
其次对于个人而言，未来一段时间的目标不再是单纯的技术进步，需要更多的与别人加强沟通，积累人脉。

回顾了一下之前的社交生活。
初中高中不细表，人设基本就是个情商很低的“学霸”，想交朋友但不太会处理人际关系的。
大学相反，成了喜欢邪门歪道的“学渣”。年年挂科，对于专业课毫无兴趣。天天吐槽山大的官僚主义和高三般的应试教育，跟大部分勤恳本份的同学格格不入。
毕业的前几年，转行跳槽技术进步，每年都会发现跟之前的同事越来越远。
最近几年，大部分时间都在研究技术。程序员之间的交流方式直来直往，社交的能力基本没啥长进，也没什么成果。

总体来说，过去的几十年里，社交这个事情对我来说确实是非常失败的。
没有什么正面的方法，倒也反思出了一些感悟：

社交是必需的的社会产物

我们变成人的一部分要求是：获取理解、思考和感受的方式。而我们“个体”的知识总是社会的产物。我们所知道的，取决于我们与他人的关系这一本质，超越个人主义并接受那些使我们成为我们现在所是的关系。通过这样做，我们就能发现我们拥有知识的局限所在，并能看到如何以那种允许我们在这个世界上更负责任地生活的方式去增加它。—-《生活的暗面》

我很赞同引言的观点，跟人保持联系，才能更加容易的了解知识局限，定时更新三观。
我以前的时候老是纠结一个问题，是跟人打交道还是跟机器打交道？最后选择了跟机器打交道，妄图来规避自己社交能力不足这个弱项。
现在回头看，压根就不是一个选择题，跟人打交道是无可避免的，也是必需的。

避免盲目无效的社交

社交手段确实有很多好处，但是现在很多人把“关系”、“社交”无限的夸大乃至神化到了一种病态的地步，大部分传授的都是溜须拍马谄媚之流。是酒桌饭后的谈资，是扯大旗吹牛的材料，唯独不是有效的社交。

避免盲目无效的社交，需要看清楚两点：

所有的人都喜欢并重视甚至偏爱一种交换——公平交换。

首先，要承认自己能力有限，有公平交换的需要，保持一个良好的心态。
我确实需要别人的帮助，我自身也有很多问题，那么在与人相处的时候自然也会待人更加的宽容。
其次，公平交换的前提是价值相等，要提升自我的价值。
再次，交换是有成本的– 时间成本，精力成本，不进行不必要的交换
最后，学会独善其身，以不给他人制造麻烦为美德，用自己的独立赢得尊重

社交是需要方法的
  • 定期清理筛选自己的圈子，比如通讯录，朋友圈
  • 放低姿态去尊重每一个人，不再以“自我”为核心，承认人与人之间的差异
  • 寻找志同道合的伙伴，使用归属感激励他人和自己
  • 从共同点出发寻找更多的共识

总体逻辑

RSS（Really Simple Syndication）服务获取信息

RSS（简易信息聚合）是一种消息来源格式规范，用以聚合经常发布更新数据的网站，例如博客文章、新闻、音频或视频的网摘。RSS文件（或称做摘要、网络摘要、或频更新，提供到频道）包含全文或是节录的文字，再加上发布者所订阅之网摘数据和授权的元数据。Really Simple Syndication“聚合真的很简单”就是RSS的英文原意,把新闻标题、摘要（Feed）、内容按照用户的要求，“送”到用户的桌面就是RSS的目的 (from wiki)

RSS文件会使用xml描述标题、摘要、内容，提供更佳简单纯粹的内容呈现、一目了然的阅读体验。

即将过去的2016，是挑战之年，也是蜕变之年。

年初前端架构组的正式组建，是公司快速发展，前端团队极速膨胀的需求。
业务逐渐增加的繁冗和复杂，前端团队需要解决的问题也愈发明显：

  • raycloud 框架在新的业务场景之下逐渐暴露出问题 (Raycloud提供hashchange router和模块定义)
    • 对于数据和视图并有没强制约束，加之jquery的频繁使用，数据层掺杂dom操作的代码屡见不鲜。
    • 为了性能考虑，使用字符串拼接，代码可读性和可维护性非常之差。
    • 没有提供完善的component机制,公用模块的种类很多

番茄工作法

事情很多，时间零碎，总是被打断
适用于工作，学习，趣味性不足的任务。

命名了一个番茄时间 – 25分钟

基本套路：
每天

1.todo list
我要干嘛。。
我还要干嘛

ps：相对合理，不要定死

2.do
25min 一个番茄时间
5min 休息时间
…
最多4个tomato

大break 15-30min

3.record
每个tomato都干嘛了

4.分析
分析时间

5.总结
一段时间的总结

工具：
番茄钟
todo list
总表

第二个套路

处理打断

坚持不了25分钟，，第一次基本会被打断10次

＝＝》25分钟不可分割，打断就没有了

打断：
内部打断：看看手机，，看看淘宝吧。。。。
内部打断怎么办？
通常都不是什么紧急的事情
so,分类，今天的，加todolist
某天做，加总表 +u
记完回去，，不算打断。

外部打断：
    最多等我25分钟，
    先记录，协商时间。（5分钟之后。。。）
紧急打断：
    推翻番茄钟，重新拟定todo list
标记打断。
正视打断，打断非常正常

休息：
休息要立马停止任务
不要想任务，不讨论任务
4个番茄中，一个大休息

分析：
分析的时间不要超过25分钟

预估能力：
如果一个todo 超过了5个tomato,需要拆分。
如果一个todo 不足一个tomato,小的废弃，大的回忆，中间的凑。

不要有负罪感，
回忆和总结。

使用electron开发桌面端

现在前端想要开发个桌面端软件，已经是件非常简单的事情了。

  • electron(atom-shell): ATOM,VSCode,
  • NW.js(node-webkit): 钉钉
  • heX:有道词典

Electron 是什么？

Electron 是一款可以利用 Web技术 开发跨平台桌面应用的框架，最初是 Github 发布的 Atom 编辑器衍生出的 Atom Shell，后更名为 Electron。
继承了Chromium + Node.js。
因此可以使用 HTML 和 CSS 来实现应用的 GUI 界面，用 JavaScript 调用丰富的原生 API 实现桌面应用。

简单点就是，以前我们开发的都是Web应用,很多东西都被浏览器给限制了,跨域，同源，http-only,不能读写文件,balabala..
如果浏览器也属于我们自己，不是就是个桌面软件啦？

为什么选择Electron？

作为一个电子系出身的前端工程师，我果断选择了electron。（开玩笑的）

  • 更新，用的人更多。
  • 性能更好。
  • 文档更加丰富。
  • 丰富的工具支持。
  • electron的更新速度特别快，最新版本的环境支持：

    Electron: 1.2.3 Node: 6.1.0 Chromium: 51.0.2704.84 V8: 5.1.281.59

Node: 6.1.0 – 这就意味着，完全支持es6,除了module那一块。

Electron 流程

  • 启动的时候调用package.json中定义的main
  • main文件会创建一个内含渲染完的web页面的应用窗口
  • 添加与你操作系统的原生GUI（图形界面）交互的功能
  • web页面定义应用内部交互
